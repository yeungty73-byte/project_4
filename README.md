A PPO-based autonomous racing agent for the AWS DeepRacer simulator, incorporating
insights from autonomous racing and reward shaping literature. Features federated
multi-worker training across PACE ICE cluster nodes.

## Algorithm Stack (v1.1.8)

The agent is a **PPO actor** (Schulman et al., 2017) with three race-type heads
(Time Trial, Object Avoidance, Head-to-Bot) sharing a **Swin-UNet++ + U-Transformer**
obs encoder (`utransformer.py`; Petit et al., 2021; Cao et al., 2021; Liu et al., 2021)
over the 38,464-dim DeepRacer sensor obs (camera 19,200 + LIDAR + parameters). The three
heads constitute a **hard-parameter-sharing multi-task architecture** (Ruder, 2017):
the shared trunk amortises representation cost across three distinct reward geometries
while per-head parameters specialise to TT, OA, and H2H respectively.

Sitting alongside PPO is an **off-policy critic ensemble** (`td3_sac_ensemble.py`, class
`TD3SACEnsemble`) that combines a deterministic-policy TD3 critic pair (Fujimoto
et al., 2018) with a stochastic SAC critic (Haarnoja et al., 2018), updated every
`_td3sac_update_freq = 4` environment steps and warm-started by behaviour cloning
(`harvest_htm_pilots` -> `pretrain_td3_bc`, 2000 BC steps). The ensemble shares a
replay buffer with the on-policy rollouts and broadcasts its twin-critic minimum back
as an auxiliary value baseline.

An **Intrinsic Curiosity Module** (`icm.py`, Pathak et al., 2017) is layered on top:
encoder dim $\phi$ in R^64, inverse-model coefficient $\beta = 0.2$, scaling $\eta = 0.01$.
The intrinsic bonus is

$$r_i = \eta * 0.5 * ||\phi(s_{t+1}) - \hat\phi(s_{t+1} | s_t, a_t)||^2 * w(seg(s_t))$$

with hotspot weight $w\ \in\ [1, 3]$ driven by a separate `failure_sampler` that
maintains a crash-density histogram over track segments and feeds it through
`icm_module.update_hotspot_weights()`. Per-episode bonus mass is logged to
TensorBoard via `writer.add_scalar('icm/ep_bonus_sum', ...)`. Burda et al.'s Random
Network Distillation is *not* used — we mention it in the report only as an
alternative intrinsic-bonus family.

The whole pipeline is gated by a **three-phase reward anneal** (A→B→C) on the
harmonised track-progress signal $p_star = 1 - \left(1 - \frac{p_raw}{100}\right)^2$ (a concave
amplifier that lifts the BSTS signal-to-noise ratio above the Kalman observation
floor) and decomposed online by a **BSTS Kalman tracker** (`bsts_seasonal.py`,
`analyze_logs.py`) with spike-and-slab feature selection. Telemetry features are
sourced from `gg_diagram.py` — the GG ellipse / friction-circle utilization, 
$\sqrt{\left(\frac{a_{lat}/}{a_{max}}\right)^2 + \left(\frac{a_{lon}}{a_{max}}\right)^2}$, trail-braking overlap, velocity profile,
and predictive curvature (Milliken & Milliken, 1995; Heilmeier et al., 2020) — which
feed directly into the BSTS regression layer.

Reward shaping itself is driven by `adaptive_reward_shaper.py` (v1.5.0b): a
multiplicative compliance multiplier in `[0.1, 1.0]` plus a `TrackProgressAnnealer`
whose phase blend is BSTS-driven, consumed by both `brake_field.py` and
`race_line_engine.py` and grounded in Ng et al. (1999) potential-based shaping
and Amodei et al. (2016) concrete-problems framing.

All figures (report + presentation) use the unified **denim theme**
(`denim_theme.py`): denim-blue family `#1B3A5C / #3A6B8C / #5B9BD5` with warm
complementary accents (amber `#D4A03C`, terra cotta `#C75B39`, sage `#6BB38A`).
Report tables mirror the same denim header band + light denim row stripes.

## Module Map

The production training graph wires six first-class modules around the PPO trunk;
each is grounded in published prior art so the contribution is the **composition**,
not any individual block.

| Module | Role + citations |
|--------|-----------------|
| `utransformer.py` | Swin-UNet++ / U-Transformer obs encoder over the 38,464-dim sensor obs — Petit (2021); Cao (2021); Liu (2021) |
| `gg_diagram.py` | Motorsport features for BSTS: GG-ellipse utilization `sqrt((a_lat/a_max)^2 + (a_lon/a_max)^2)`, trail-braking overlap, velocity profile, predictive curvature — Brach (2011); Milliken (1995); Heilmeier (2020); Kapania (2016) |
| `adaptive_reward_shaper.py` | Multiplicative compliance multiplier in `[0.1, 1.0]` + `TrackProgressAnnealer` with BSTS-driven phase blending — Ng (1999); Amodei (2016) |
| `context_aware_agent.py` | Race-type-conditioned PPO actor head gating TT / OA / H2H channels on a single shared trunk (**multi-task hard-parameter-sharing** — Ruder, 2017; Schulman, 2017; Bengio, 2009) |
| `federated_pool.py` | Federated checkpoint pooling across PACE ICE workers; supports the off-policy ensemble's replay during distributed runs — Fujimoto (2018); Haarnoja (2018) |
| `stuck_tracker.py`, `htm_reference.py` | Engineering scaffolding — stuck-state detection and HTM heatmap reference. Not novel claims. |

The architectural contribution is the composition: (a) Petit/Cao Swin-UNet++ obs
encoder, (b) Brach/Milliken motorsport features, (c) Ng-1999-grounded multiplicative
shaper, and (d) Pathak/Fujimoto/Haarnoja off-policy ensemble — gated through a single
curriculum schedule audited end-to-end with Brodersen-Scott-Varian BSTS.

## Apptainer (Singularity) sim setup — troubleshooting

If `start_deepracer.sh` exits with:

- `FATAL: Image file already exists: "deepracer_base.sif" - will not overwrite`
- `FATAL: Unable to build from deepracer.def: … no such file or directory`
- `FATAL: While checking container encryption: could not open image .../project_4/deepracer.sif`

then the script is falling through the pull/build paths and trying to start an instance
from a non-existent local `deepracer.sif`. The working images normally live one directory
up. Fix:

```bash
cd ~/scratch/P4_deepracer/project_4
rm -f deepracer_base.sif                      # remove stale partial
ln -sf ../deepracer.sif deepracer.sif          # symlink working derived image
ln -sf ../deepracer_base.sif deepracer_base.sif
bash start_deepracer.sh                        # pull/build FATALs are benign
```

Verify:

```bash
apptainer instance list          # should show: deepracer <PID> <IP> .../deepracer.sif
apptainer exec instance://deepracer pgrep -af gzserver   # Gazebo process running
apptainer exec instance://deepracer pgrep -af rollout    # markov.rollout_worker running
```

**Important:** `run.py` in this folder is a standalone PPO loop that does **not**
connect to the DeepRacer sim. Real training happens via `markov.rollout_worker` inside
the container (started automatically by `start_deepracer.sh`). The `[GUARD]` log line
introduced in v201 will report `JSONL is EMPTY` if `run.py` is mistakenly run on a
node without the sim wiring.

## v202 — Gym bridge port discovery (Apptainer)

The ZMQ client used to be hardcoded to `tcp://127.0.0.1:8888`. Under Apptainer there
is no `-p 8888:8888` port-map (unlike Docker), so the sim's ZMQ REP socket binds on a
different ephemeral port in the host network namespace and the Python client silently
failed — PPO ran for hours on an environment whose `terminated` never went True.

### Fix

1. `packages/deepracer_gym/zmq_client.py` now reads `GYM_HOST`/`GYM_PORT` from env.
2. `env_for_client.sh` exports the user-hashed ports (same scheme as `start_deepracer.sh`).
3. `run.py` (v202) does a TCP preflight on `$GYM_HOST:$GYM_PORT` and aborts with a
   clear message if unreachable — no more 100k stub-episode runs.

### Workflow

```bash
bash start_deepracer.sh            # bring up Apptainer sim (one time per session)
ss -ltnp | grep rollout_worker     # find actual ZMQ REP port of markov.rollout_worker
export GYM_PORT=<that port>        # or: source ./env_for_client.sh
python run.py --environment deepracer-v0 --results_dir results/live
```

If the preflight aborts, the rollout_worker is not listening. Check
`pgrep -fa rollout_worker` and the sim's start log under
`~/scratch/P4_deepracer/apptainer/`.

### v202b — auto-discover GYM_PORT

`env_for_client.sh` now queries `pgrep markov.rollout_worker` + `ss -ltnp` to find the
actual ZMQ REP port the running rollout_worker is listening on, and exports it as
`GYM_PORT`. Falls back to the hashed port if no rollout_worker is found.

Also fixed `run.py` preflight `SystemExit` path (prior `import sys` inside the
conditional shadowed the module-level `sys` reference via Python's local-name binding
rules).

### Known remaining issue — ZMQ protocol mismatch

Even after a successful TCP preflight and `env.reset() succeeded on attempt 1`, the
JSONL stays 0 bytes: `terminated` never flips True. Direct probe:
`socket.REQ -> msgpack.packb({'ready':1}) -> recv()` times out. The
`markov.rollout_worker` inside the Apptainer image expects a different message schema
than what `packages/deepracer_gym/zmq_client.py` sends. Next step: inspect
`/opt/amazon/markov/rollout_worker*.py` inside the container to align request keys.

**What changed.** Prior waves generated per-figure synthetic data ad hoc, creating
plot-to-prose incoherence. Wave 5 replaces all of that with a **single synthetic
substrate per race type** (`synth_substrate.py`, seed = 42), which must clear **four
validation gates** before a single figure is accepted:

1. Phase-A/Phase-C progress variance ratio >= 2.5
2. Period-60 seasonal amplitude >= 0.5 × A_60 recovered by Phase-C OLS
3. CUSUM `k_star` inside the Phase-B window `[p_A, p_B]`
4. Lap-time identity `L/(v_bar_c * p) + drag = T_lap` closes to machine precision
   (`L = 16.635 m`)

The trailing-5 eval stats are reported *as produced by the seed-42 substrate* — no
post-hoc recentering of any kind.
